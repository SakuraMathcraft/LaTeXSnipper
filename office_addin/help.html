LaTeXSnipper Word 加载项

把 LaTeX 转为 Word 可编辑的 OMML 公式，并为由本加载项创建的公式保存源代码和编号信息。

使用前提：启动桌面端 LaTeXSnipper Office Bridge；Word 文档必须允许加载项读写内容。 Word 功能同时要求 WordApi 1.3 与 SharedRuntime 1.1； 共享运行时让 Ribbon 命令与侧边栏在同一长期运行的任务窗格中处理编辑状态。

可支持的 Word 版本

环境本加载项状态依据与说明
Word for Windows 桌面版 支持目标平台 需同时支持两个 requirement set；其中限制更高的 SharedRuntime 1.1 要求 Microsoft 365 Word Version 2205 (Build 15202.10000) 或 Word 2024。Word 2016、2019 与 2021 不在本加载项的 Windows 支持范围内。
Word for Mac 桌面版 API 可支持，需实机回归 限制更高的 SharedRuntime 1.1 最低要求 Word 16.61 (22040100)。本项目的主要手动验证环境目前仍是 Windows 桌面版。
Word on the web 不作为当前支持平台 Microsoft 表中支持这两个 requirement set，但本加载项依赖本机 Bridge 与桌面 OCR 工作流，未承诺网页版连通性与交互稳定性。
Word on iPad / 移动端 不支持 SharedRuntime 1.1 不支持 Office on iOS；本机 Bridge 流程也没有为移动端交付。

官方参考： Word JavaScript API requirement sets、 Shared runtime requirement sets、 Dialog API。

公式对象的界限

“LaTeXSnipper 公式”不是任意 Word 公式。它必须是本加载项插入的 OMML 内容控件，带有 latexsnipper-eq-{id} 标签，并在文档设置中保存对应 LaTeX 源代码。

  • Word 自带“插入公式”、MathType 或从外部复制的原生公式不能直接 Load、Delete Selected 或自动编号。
  • 如果用户手动删除了内容控件标签或破坏了编号表格，加载项不会猜测其来源或自动修复。
  • 带编号公式使用无边框三列表格：左右留白对称，中间公式居中，右侧显示编号。

Ribbon 功能说明

Editor

打开完整公式编辑器对话框，提供 MathLive 可视输入、LaTeX 源代码和符号面板。仅打开编辑器不会修改文档；只有执行 Insert 或在加载后的公式上执行 Update 才写入 Word。

Insert Formula

将侧边栏当前 LaTeX 通过 Bridge 转成 OMML，在光标位置插入新的 LaTeXSnipper 公式。

  • Display 勾选时以独立居中段落插入；未勾选时按行内公式插入。
  • Auto number 与 Manual # 互斥；设置编号后公式自动变为显示公式。
  • 不能在任何已有 LaTeXSnipper 公式对象内部继续插入；对于带编号公式，其布局表格与编号后的单元格边界同样属于不可插入区域。
  • 连续插入编号公式时，加载项会保留公式之间的分隔段落，确保每个编号公式仍是可独立删除和编辑的对象。
  • Insert 始终创建新对象，不会覆盖当前选中的既有公式。

Screenshot OCR

等待桌面端下一次截图识别，并把识别得到的 LaTeX 填入编辑器。此功能只填入输入框，不会自动插入或更新文档；用户可以先校正结果再 Insert。

Load Selected

读取当前选中的 LaTeXSnipper 公式源代码，并在对话框中进入编辑状态。选择公式本体、其编号或整个编号表格均可定位同一公式；未选中本加载项公式时不会载入其他内容。

Delete Selected

直接删除当前选中的 LaTeXSnipper 公式，不要求先 Load。无编号公式仅删除其内容控件及公式内容，即使它位于用户自己的表格中也不会删除该表格；带编号公式删除自身布局行/表格和编号；两者都会清理该公式的文档设置元数据。

Auto Numbered

把一个已选中的、尚未编号的 LaTeXSnipper 公式转换为自动编号显示公式。它不创建新公式，不作用于 Word 原生公式，也不会将手动编号强制改为自动编号。

Renumber All

按文档顺序重新编号所有设置为自动编号的 LaTeXSnipper 公式，并统一其表格对齐。手动编号公式完全保留原值，不参加序列，也不会被此命令覆盖。

加载后更新

通过 Load Selected 打开的编辑器中点击 Update，加载项会使用同一公式 ID 重建带标签的 OMML 容器，并保存新的 LaTeX 源代码。

  • 修改是否编号会改变文档结构：无编号为段落内容控件，带编号为三列表格。
  • 保持编号状态时，Update 只改写当前公式单元格和当前编号，不会替换同表格中的其他编号行。
  • 自动编号被删除或改为手动编号后，其余自动编号公式会重新排号。
  • 更新只针对已加载的 LaTeXSnipper 公式；它不是对任意选区的通用 Word 公式替换器。

数据与隐私边界

  • Bridge URL、令牌和每条公式的 LaTeX/编号记录存储在当前 Word 文档的 Office settings 中。
  • LaTeX 转换和截图 OCR 默认发送给本机 Bridge；如果修改 Bridge URL，则数据会发送到用户配置的地址。
  • 复制仅包含渲染结果而不包含文档设置的内容到其他文档时，新文档无法 Load 其源代码。

当前能力边界

已覆盖：Office.js 可稳定实现的核心 Word 流程：共享运行时 Ribbon 命令、标签化 OMML 插入、源代码持久化、按选择载入和更新、自动/手动编号、重编号、删除及对话框交互。
尚未达到“完全完善”：Word JavaScript API 不提供“把任意原生公式还原为原始 LaTeX”的可靠接口，因此本加载项只能编辑自身创建且元数据仍存在的公式。此外，跨文档元数据迁移、损坏对象修复、共同编辑冲突处理、Mac 实机完整回归和 Word Web 交付仍未完成。

结论：当前实现已选择了 Office.js 下正确的可编辑公式架构，但还没有达到产品层面的能力边界或发布级完备程度。继续提升的重点应是实机自动化/回归验证、损坏检测与恢复、跨文档导入策略和跨平台测试，而不是再为未知对象添加静默兜底。
