# LaTeXSnipper v2.3.1 双语阅读功能技术方案

## 1. 目标

在主窗口新增 `双语阅读` 入口，位置放在 `数学工作台` 按钮前面，提供一个独立的 PDF 中英文对照阅读窗口。

该功能优先复用现有 `PDF 预览 / 放大镜 / 适应窗口 / 适应宽度 / 网格预览 / 后端切换` 能力，避免重复建设 PDF 渲染链路。

`v2.3.1` 的目标不是一次性做成“精确段落坐标级双语阅读器”，而是先落地一个可维护、可扩展、可发布的第一页版本。

---

## 2. 版本范围

### 2.1 v2.3.1 必做

1. 主窗口新增 `双语阅读` 入口按钮。
2. 打开独立的双语阅读窗口。
3. 左侧复用现有 PDF 预览能力。
4. 右侧显示当前 PDF 页对应的中文译文。
5. 支持切换 PDF 预览后端、适应窗口、适应宽度、适应文本宽度、放大镜。
6. 支持选择翻译后端。
7. 默认接入免费本地翻译后端，其他 API 由用户自行配置。

### 2.2 v2.3.1 不做

1. 不做 PDF 块级坐标到译文段落的精确双向高亮。
2. 不做公式块、表格块、图注块的独立翻译规则。
3. 不做复杂双栏论文的精细阅读顺序纠正。
4. 不做整篇文档的专业术语库与记忆库。
5. 不做边读边缓存到数据库的长期知识管理。

---

## 3. 入口设计

### 3.1 主窗口入口位置

在主窗口右侧编辑区顶部工具栏中，新增 `双语阅读` 按钮，放在 `数学工作台` 前面。

建议顺序：

`图片识别` → `PDF识别` → `手写识别` → `复制` → `导出` → `双语阅读` → `数学工作台`

原因：

1. `双语阅读` 属于 PDF/文档工作流，不应放到自动排版窗口内。
2. 它是独立主功能，和数学工作台同级。
3. 位置靠近 `PDF识别` 和 `数学工作台`，符合用户心智路径。

## 4. 复用策略

### 4.1 直接复用的现有模块

直接复用：

- `src/handwriting/pdf_view_fitz.py`
- `src/handwriting/pdf_view_poppler.py`

可直接继承或组合使用的能力：

1. PDF 加载与翻页显示
2. 放大镜
3. 实际大小
4. 适应窗口
5. 适应宽度
6. 适应文本宽度
7. 网格预览
8. Poppler / Fitz 后端切换

#### B. 预览窗口工具栏逻辑

优先复用或抽取自：

- `src/handwriting/document_preview_window.py`

可复用逻辑：

1. PDF 后端状态展示
2. 工具栏按钮接线
3. 网格布局切换
4. 预览区域重建逻辑

#### C. PDF 文本提取能力

优先复用现有依赖链：

- `PyMuPDF` 已在项目 PDF 相关流程中使用

`v2.3.1` 先用 `PyMuPDF` 做逐页文本提取，不引入新的重型 PDF 解析框架。

## 5. 新增窗口设计

### 5.1 新窗口职责

新增独立窗口，例如：

- `src/reader/bilingual_pdf_window.py`

该窗口负责：

1. 打开 PDF
2. 显示左侧 PDF 预览
3. 提取当前页文本
4. 调用翻译后端
5. 在右侧显示当前页中文译文

该窗口不负责：

1. TeX 排版
2. LaTeX 编译
3. 数学工作台计算

### 5.2 界面结构

建议使用左右分栏：

左侧：

- PDF 预览区
- 复用现有放大镜和缩放逻辑

右侧：

- 当前页原文摘要或原文纯文本
- 当前页中文译文
- 翻译状态与翻译引擎选择

顶部工具栏：

1. 打开 PDF
2. 翻译引擎选择
3. 实际大小
4. 适应窗口
5. 适应宽度
6. 适应文本宽度
7. 网格选择
8. 重新翻译当前页
9. 批量翻译全部页

## 6. 翻译引擎策略

### 6.1 默认后端

默认接入本地免费后端：

- `Argos Translate`

原因：

1. 免费
2. 离线
3. 适合作为“默认可用能力”
4. 不依赖用户先申请 API key

### 6.2 可选后端

其他翻译引擎走用户配置：

1. DeepL API
2. Google Cloud Translation
3. OpenAI 兼容接口
4. 现有外部模型接口

### 6.3 后端抽象

新增统一接口层，例如：

- `src/backend/translation/engine.py`
- `src/backend/translation/argos_engine.py`
- `src/backend/translation/external_api_engine.py`

统一接口建议：

```python
class TranslationEngine:
    def is_available(self) -> bool: ...
    def translate_text(self, text: str, source_lang: str, target_lang: str) -> str: ...
    def display_name(self) -> str: ...
```

这样后续切换翻译后端不影响阅读窗口逻辑。

## 7. 文本处理策略

### 7.1 v2.3.1 页级对照

第一页版本只做 `页级对照`：

1. 提取当前页文本
2. 清洗页眉页脚和空白行
3. 翻译为中文
4. 右侧显示当前页中文结果

优点：

1. 落地快
2. 结构简单
3. 可直接复用当前 PDF 预览能力
4. 风险低

### 7.2 暂不做块级对齐

块级对齐涉及：

1. 版面阅读顺序
2. 双栏分页
3. 公式与正文混排
4. 图表说明文字

这一层复杂度明显高于 v2.3.1，建议延期到后续版本。

## 9. 落地阶段划分

### 阶段一：主窗口入口与阅读窗口骨架

1. 主窗口增加 `双语阅读` 按钮
2. 能打开 PDF
3. 左侧显示 PDF
4. 右侧先显示原文文本

### 阶段二：默认翻译后端接入

1. 集成 `Argos Translate`
2. 支持页级翻译
3. 支持手动刷新当前页翻译

### 阶段三：用户自配 API

1. 设置页增加翻译引擎配置
2. 支持 DeepL / Google / 外部模型
3. 增加优先级与回退策略

## 10. 风险与约束

### 10.1 技术风险

1. PDF 文本抽取顺序不稳定
2. 双栏论文文本顺序可能错乱
3. 公式混排页翻译质量可能不稳定
4. 本地免费翻译质量弱于商业 API

### 10.2 控制策略

1. `v2.3.1` 只承诺页级对照，不承诺块级精确对齐
2. 先支持“读得懂”，再优化“对得准”
3. 翻译后端做成可切换策略，避免被单一后端绑定

---

## 11. v2.3.1 更新点建议文案

### 用户可见更新

1. 主窗口新增 `双语阅读` 入口，位于 `数学工作台` 前。
2. 新增 PDF 中英对照阅读窗口。
3. 复用现有 PDF 预览能力，支持放大镜、适应窗口、适应宽度和网格预览。
4. 默认接入免费本地翻译后端，支持开箱即用的页级中英对照阅读。
5. 预留外部翻译 API 配置扩展能力。

### 内部实现更新

1. 抽取可复用 PDF 预览装配逻辑。
2. 增加统一翻译引擎抽象层。
3. 阅读窗口采用页级文本抽取与页级翻译策略，降低首版复杂度。

---

## 12. 结论

`双语阅读` 应作为主窗口中的独立入口落地，而不是附着在自动排版窗口内部。

`v2.3.1` 的正确方向是：

1. 入口独立
2. PDF 预览能力复用
3. 默认翻译后端可用
4. 页级对照优先
5. 后续再逐步升级到段落级和坐标级对齐

这条路线复用率高、边界清晰、维护成本可控，适合作为 `v2.3.1` 的新增功能点。
